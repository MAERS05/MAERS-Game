# sq-du 架构说明（含扩展指南）

## 1. 分层总览

### 1.1 引擎层：`base/engine.js`
职责：
- 回合状态机推进（开始 / 装配 / 决策 / 暴露 / 重筹 / 行动 / 结算 / 结束）
- 统一派发阶段事件
- 调用时机层与裁判层
- 对外提供 API（UI / AI / 联机驱动）

不负责：
- 不直接写某个效果具体规则（如“创伤如何扣血”）

---

### 1.2 时机层：`sq-du/effect/timing.js`
职责：
- 维护待触发效果队列 `pendingEffects`
- 按阶段事件分发到期效果
- 作为“时机到达 -> 触发效果”的唯一入口

不负责：
- 不描述效果本体数值含义
- 不决定技能如何附加效果

---

### 1.3 技能层：`sq-du/skill/*/*.js`
职责：
- 只负责“在什么条件下、什么时机，给自己或对方附加什么效果”
- 负责前置 / 后置 / 阶段挂载的条件判断
- 通过 `EffectLayer.queueEffect(...)` 或 `queueDelayedEffect(...)` 附加效果

不负责：
- 不直接实现效果本体数值
- 不负责效果触发后的具体数值结算

---

### 1.4 效果层：`sq-du/effect/*.js`
职责：
- 只保留效果本体
- 定义效果名称、描述、适用对象和触发实现
- 在到达对应时机时执行具体效果

不负责：
- 不负责技能条件判断
- 不负责队列调度

---

### 1.5 裁判层：`main/judge.js` + `base/resolver.js`
职责：
- 计算 clash（交锋情形）
- 推演伤害与处决
- 产出 `ResolveResult`

不负责：
- 不调度效果触发策略

---

### 1.6 表现层：`sq-du/ui.js`
职责：
- 订阅引擎事件并渲染
- 转发用户输入到引擎 API

不负责：
- 不改写规则状态

---

## 2. 事件机制与时机

### 2.1 阶段事件
- `TURN_START_PHASE / TURN_END_PHASE`
- `EQUIP_START / EQUIP_END`
- `DECISION_START / DECISION_END`
- `EXPOSE_START / EXPOSE_END`
- `REDECIDE_START / REDECIDE_END`
- `ACTION_START / ACTION_END`
- `RESOLVE_START / RESOLVE_END`

### 2.2 统一时机分发层
- `EffectTimingLayer.dispatch(...)`
- 作用：扫描 `pendingEffects`，到点后调用对应效果的 `onPhase`

### 2.3 统一效果队列
- `EffectLayer.queueEffect(...)`
- `EffectLayer.queueDelayedEffect(...)`
- 含义：技能把效果排入队列，时机层到点触发

### 2.4 阶段后快照同步
- `PHASE_STATE_SYNC`
- 含义：每次阶段效果执行后同步一次状态快照，避免“逻辑触发了但 UI 还没刷新”。

---

## 3. 三层数据流（推荐理解路径）
1. 技能在某个条件满足时，把效果挂入队列
2. 引擎触发阶段事件
3. 时机层判断是否到期并调用效果
4. 裁判层产出 `ResolveResult`
5. 技能或效果按结果继续挂后续效果
6. 引擎写回状态并推进下一阶段

---

## 4. 成功/失败判定收拢

已收拢到 `main/effect.js` 的统一入口：
- 先读基础结算（`clash`）
- 再读最终伤害（效果改写后）
- 输出语义判定（`success/reason`）

用途：
- 解决“侥幸仍触发命中类效果”等问题
- 方便未来扩展免疫、护盾、转伤、一次性抵消等新规则

---

## 5. 目录约定

### 5.1 技能目录
- `sq-du/skill/attack/*.js`
- `sq-du/skill/guard/*.js`
- `sq-du/skill/dodge/*.js`

约定：
- 只写技能条件与附加动作
- 文案里的描述保持为技能语义来源
- 不直接实现效果本体

### 5.2 效果目录
- `sq-du/effect/*.js`

约定：
- 只写效果本体
- 效果的触发语义和数值含义集中在这里

### 5.3 时机目录
- `sq-du/effect/timing.js`
- `sq-du/effect/timing-constants.js`
- `sq-du/effect/timing-registry.js`

约定：
- 只处理“何时触发”
- 不写具体效果业务

---

## 6. 如何扩展（不重构底层）

### 6.1 新增技能
步骤：
1. 新建 `sq-du/skill/xxx/your_skill.js`
2. 根据条件决定挂载哪个效果
3. 使用 `EffectLayer.queueEffect(...)` 或 `queueDelayedEffect(...)`
4. 不在技能里实现效果本体

建议：
- 技能的描述与逻辑应保持一致
- 附加效果尽量走统一队列

---

### 6.2 新增效果
步骤：
1. 新建 `sq-du/effect/your_effect.js`
2. 实现效果本体
3. 在 `constants.js` 注册 `EffectId` + `EffectDefs`
4. 在 `effect-handlers.js` 注册 handler

建议：
- 效果文件只表达“效果是什么”，不要混入技能条件

---

### 6.3 新增时机
步骤：
1. 新增时机常量
2. 在 `EffectTimingLayer` 中支持该时机
3. 让技能把效果挂到该时机
4. 让效果在 `onPhase` 中处理该时机

---

### 6.4 新增判定方式（例如“免疫一次攻击”）
不需要推翻架构，按以下做：
1. 在效果执行时写入状态标记或改写伤害
2. 扩展统一判定入口（攻击是否成功）
3. 后置效果依赖统一判定，不再自己散判

---

### 6.5 新增模式（联机 / 脚本）
- 复用 engine + effect + resolver
- 替换输入驱动
- UI 层无须承载规则差异

---

## 7. 职责边界检查清单（改动前后必看）
- 是否把技能条件写进了效果文件？（应避免）
- 是否把效果本体写进了技能文件？（应避免）
- 是否把触发时机写进了效果本体？（应避免）
- 是否把规则写进了 engine？（应避免）
- 是否在 ui 中直接改游戏状态？（应避免）
- 是否存在同一效果多处触发？（要收敛）
- 是否存在“逻辑时机”和“显示时机”错位？（需 `PHASE_STATE_SYNC`）

---

## 8. 现阶段约定（当前项目）
- 技能负责附加效果
- 时机层负责触发效果
- 效果文件负责效果本体
- 结算判定用于定义“成功 / 失败”
- 阶段接口均可承载效果并在阶段后刷新状态
