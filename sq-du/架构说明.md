# 技术架构说明 (Engine Architecture)

`sq-du` 目录下的引擎系统被严格分层，保证了复杂博弈游戏的扩展性与易测性，整体遵循“逻辑与表现分离、状态修改流水线化”的设计哲学。主要包含四大模块结构：

## 1. 裁判层 `main/judge.js` (JudgeLayer)
**定位**：纯粹的物理与规则核心（无状态的纯函数集合）。
- **职责**：不拥有任何持续性状态。它仅仅接收外部提供的一个特定「时间截面快照」，然后在内部铺设一条按动速降序的虚拟时间线（Timeline），并模拟出两人发生物理碰撞的绝对客观结果（造成了多少点数伤害、谁被谁格挡、谁成功闪避等）。
- **特点**：保证绝对公平与可回溯。所有胜负判定、同归于尽、自杀处决逻辑均收敛于此，输出唯一的封包协议 `ResolveResult`。

## 2. 效果层 `main/effect.js` (EffectLayer)
**定位**：战术状态缓冲带与能力修饰器。
- **职责**：分为两大宏观生命周期：
  - `onPre`（裁决前的战前修饰）：把盘踞在玩家身上的各个 Effect（如锁链禁止闪避、力量加成伤害），编译重写成能丢给法官（Judge）判定的最终提交订单 `ActionCtx`。
  - `onPost`（裁决后的因果应用）：在法官判定出实质伤害后，触发特定被动（例如挨打就掉生命上限、或者受伤后叠加怒气），最终处理回合末机制（效果衰减、计时器递增）。
- **特点**：提供了极致的扩展能力。只需将新的规则配置注册进 `combat-identity.js` 等功能字典，即可插拔式实现各种奇幻的战斗特质，而不需要改动引擎核心。

## 3. 总线与解析枢纽 `base/resolver.js` (Resolver)
**定位**：调度者与数据路由管道。
- **职责**：如同一位制片人，串联起 EffectLayer 和 JudgeLayer。它负责从 Engine 收到两名玩家的意图回合开始处理：先丢给 Effect 进行 `onPre` 修改 → 捕捉增量（Flash Data）并塞入快照给 Judge → 接受 Judge 产出的 Timeline → 再丢给 Effect 进行 `onPost` → 将处理完毕附带上最终血量/精力计算的结果发送回 Engine。
- **特点**：处理特殊中断信号。如短路型的 `INSIGHT_CLASH`（识破废除回合）逻辑就在此判断并绕过常规战斗演算。

## 4. 引擎核心机 `base/engine.js` (Engine Controller)
**定位**：带状态的主从控制核心（State Machine Controller）。
- **职责**：维护整局游戏唯一且客观的主舞台（PlayerState）。包含回合滴答时钟（Tick Timer）、阶段状态机（配置期装备槽、三十秒死线、暴露期与结算动画期）。
- **特点**：提供所有供 UI 乃至外部（如网络模式）调用的动作 API（如提交攻击、使用洞察）。所有生命周期通过强类型的事件总线 `EventBus` 广播（Emit）给外部表现层。

## 5. UI 与前端呈现 `main/ui.js` / `.html`
**定位**：响应式视图。
- **职责**：它不干涉游戏逻辑，只倾听 `EventBus` 发来的消息，利用 `DOM` API 进行面板重绘。包含浮动的状态显示框、交手日志、三段式详细 tooltip 面板等。
- **特点**：能轻松实现前端录像回放与战局恢复（通过给它灌入历史事件流和状态快照）。
